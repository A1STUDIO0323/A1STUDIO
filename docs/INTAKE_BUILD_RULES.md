# 홈페이지 자동 제작 절대지침 (Intake Build Absolute Rules)

> `/intake` 의뢰를 받아 신규 홈페이지를 자동 생성할 때 **반드시** 준수해야 하는 통합 규칙.
> 이 문서의 모든 항목은 **절대지침**이며, 어떤 단계·어떤 티어에서도 예외 없이 적용된다.

---

## 0. 적용 범위

- A1STUDIO 코드베이스를 **클론·리브랜딩**하여 다른 회사 홈페이지를 만드는 모든 작업
- 신규 Supabase 프로젝트 생성, 새 도메인 배포, 새 PG 계정 연동 포함
- 빌드 결과물은 항상 **운영자에게 인계 가능한 상태**여야 함 (env, 운영 매뉴얼, 관리자 계정 포함)

---

## 1. 로깅 절대지침 (★ 최우선)

### 1-1. 모든 구현에 추적·조치 가능한 로그 포함
- 단순한 `console.log("error")` 금지
- 로그 한 줄에 반드시 포함: **[모듈명] [동작] [식별자(userId/reservationId 등)] [결과/에러]**
- 예시: `[Payment][REQUEST] reservationId=abc123 amount=50000 method=kakaopay`

### 1-2. 결제·예약은 시작·성공·실패 3단계 로그 필수
모든 결제/예약 흐름은 다음 3단계 로그를 **반드시** 남긴다.

```ts
console.log(`[Payment][START] userId=${u} amount=${a} method=${m} at=${ts}`);
// ... 처리 ...
console.log(`[Payment][SUCCESS] paymentId=${p} tid=${tid} at=${ts}`);
// 또는
console.error(`[Payment][FAIL] reason=${r} code=${c} at=${ts} stack=${e.stack}`);
```

대상 흐름:
- 결제 요청 / 승인 / 환불 / 부분환불 / 재시도
- 예약 생성 / 확정 / 취소 / 시간변경
- 회원가입 / 로그인 / 비밀번호 변경 / OAuth 콜백
- 알림 발송 (이메일·SMS·알림톡) 시작/성공/실패
- 외부 API 호출 (PG, 솔라피, 카카오 등) 요청/응답/타임아웃

### 1-3. 에러 메시지는 운영자가 즉시 조치 가능해야 함
- ❌ `"오류가 발생했습니다"` 금지
- ✅ `"카카오페이 승인 실패: TID=T0001 응답코드=-780 (가맹점 한도초과). 카카오페이 콘솔 > 한도 설정 확인 필요"` 형태

### 1-4. 민감정보 로그 금지
- 비밀번호, 카드번호 전체, CVV, 주민번호, 토큰 원문 → **절대 로그 금지**
- 식별 필요 시 마스킹: `card=****-****-****-1234`, `token=tok_xxxxx...abc`

## 2. DB 접근 분리 절대지침

### 2-1. Prisma / Supabase 혼용 금지
- **인증 (Auth)**: Supabase 클라이언트만 사용 (`supabase.auth.*`)
- **비즈니스 로직 CRUD**: Prisma 클라이언트만 사용
- 한 파일/라우트에 두 클라이언트를 동시에 import 금지
- 단, 인증 미들웨어에서 user 식별 후 → 별도 service 함수에서 Prisma로 작업하는 것은 가능 (계층 분리)

### 2-2. RLS 정책 활성화
- 모든 Supabase 테이블에 RLS(Row Level Security) **반드시 활성화**
- 익명 접근이 필요한 테이블도 명시적 policy로 허용

### 2-3. prisma db push 금지 (운영 환경)
- 신규 프로젝트 초기 구축 시에만 `db push` 허용
- 운영 중인 DB에는 반드시 **migration** 파일로 적용
- 카카오페이 심사 중인 DB는 어떤 변경도 금지 (아래 4-3 참조)

## 3. 회원 권한 절대지침

### 3-1. 신규 사용자는 무조건 MEMBER
- 가입 시 자동 권한 부여 금지
- 관리자(STUDIO_ADMIN/CM 등) 권한은 **관리자 UI에서 수동 승격만**
- 자동 승격 로직, 이메일 도메인 기반 승격, 첫 가입자 자동 ADMIN 등 **모두 금지**

### 3-2. 권한 체크는 서버에서
- 클라이언트의 `role` 변수에 의존한 분기 금지 (UI 표시 제외)
- 관리자 API는 반드시 서버 측에서 `role === 'STUDIO_ADMIN'` 등을 재검증

## 4. 결제 시스템 절대지침 (4단계 이상)

### 4-1. 통신판매업 신고 필수
- 4단계 빌드 시작 전 **통신판매업 신고증** 확인
- 미신고 상태면 신고 안내 후 1~3단계로 우선 빌드, 결제 모듈은 따로 분리

### 4-2. PG 심사 절차 안내
- 카카오페이 / 토스페이먼츠 등은 심사 기간이 2주~1개월 소요
- 심사 신청 가이드를 **사용자에게 명시적으로 제공** (URL, 필요 서류 리스트)
- 심사 완료 전까지는 결제 라우트를 비활성화 (placeholder 페이지)

### 4-3. ★ 카카오페이 심사 중 프로젝트는 결제 코드 절대 변경 금지
- `payments` / `reservations` / `party_reservations` 테이블
- `kakaopay_*` 컬럼
- 결제 라우트 (`/api/payments/**`, `/api/kakaopay/**`)
- 위 항목은 심사 통과 전까지 **읽기만 가능, 수정 절대 금지**

### 4-4. 환불 정책은 코드와 화면 모두에 명시
- 전자상거래법상 환불 정책 명시 의무 → 푸터·이용약관·결제 직전 페이지 3곳에 노출
- 환불 정책 데이터는 환경변수가 아닌 **DB 또는 별도 설정 파일**로 관리 (운영자 수정 가능)

### 4-5. ★ 카카오페이 심사 필수 페이지 — `/privacy`, `/terms`
- **개인정보처리방침 (`/privacy`)** 와 **이용약관 (`/terms`)** 페이지는 4단계 빌드 시 **반드시 사전 구축** (심사 거절 사유 1순위)
- 푸터에서 두 페이지로 가는 링크가 **명확히 노출**되어야 함
- 두 페이지는 placeholder 금지 — 실제 사업자 정보·환불·분쟁 처리 절차를 채워서 제출
- 카카오페이 심사 제출 직전 체크리스트:
  - [ ] `/privacy` 접속 가능 + 사업자 정보 일치
  - [ ] `/terms` 접속 가능 + 환불 규정 명시
  - [ ] 푸터에서 두 페이지 링크 1클릭 도달

### 4-6. ★★ 푸터 사업자 정보 표기 절대지침 (전 단계 공통, 4단계 필수)
> 카카오페이 심사 실측 거절 사례: "사업자등록번호·대표자명 확인 불가 / 기재 주소가 사업자등록증과 상이"
> → 푸터에 아래 5개 항목이 **사업자등록증과 100% 일치**해야 함.

**필수 표기 항목 (5개, 모두 누락 금지):**
1. **상호명** (법인등록부/사업자등록증 정식 명칭, 영문 병기 가능 — 예: `에이원스튜디오(A1STUDIO)`)
2. **대표자명**
3. **사업자등록번호** (`XXX-XX-XXXXX` 하이픈 포함 형식)
4. **사업장 주소** (사업자등록증 주소와 완전 일치 — 도로명/지번, 동·층·호수까지)
5. **대표 전화번호**

**권장 표기 (4단계 결제 시 필수):**
- 통신판매업 신고번호
- 개인정보책임자 (이름 + 이메일)
- `/privacy`, `/terms` 링크

**구현 규칙:**
- 푸터 컴포넌트는 **모든 페이지 공통 레이아웃**에 포함 (홈, 마이페이지, 결제 페이지 등 예외 없음)
- 사업자 정보는 **하드코딩 금지** — `.env` 또는 `lib/business-info.ts` 단일 소스에서 import
- 빌드 완료 시 사업자등록증 사본과 푸터 표기를 **육안 대조 검수** 필수
- 참조 이미지: A1STUDIO 푸터 (상호/대표자명/사업자등록번호/주소/전화 5행 정렬)

## 8. 코드 품질 절대지침

### 8-1. 불필요한 추상화·방어 코드 금지
- 발생하지 않는 시나리오에 대한 try-catch 금지
- 사용처가 1곳뿐인 유틸 함수 분리 금지
- 미래의 가능성을 위한 옵션·플래그 금지

### 8-2. 주석은 WHY가 비자명할 때만
- WHAT은 코드와 식별자로 표현
- 현재 작업 컨텍스트(`// 영업 요청으로 추가`) 금지
- 제거해도 혼란 없으면 쓰지 말 것

### 8-3. 보안 취약점 즉시 수정
- OWASP Top 10 (SQL 인젝션, XSS, CSRF 등) 발견 시 즉시 수정
- 발견 시 사용자에게 보고

## 9. 솔라피·알림 절대지침

### 9-1. 예약 SMS는 시작/성공/실패 카운팅
- 솔라피 예약 SMS는 발송 시도 시 카운트 증가
- 실패 시에도 별도 카운터로 추적
- 운영자가 발송 한도를 항상 파악 가능해야 함

### 9-2. 장기대관 안내문 자동 발송
- 등록 시: 요금안내문 즉시 발송
- 매 이용일 KST 10시: 이용안내문 솔라피 예약 발송
- 이 흐름은 빌드 시 빠짐없이 포함 (3단계 이상)

### 9-3. 장기대관 고객 신청 흐름 (공개 라우트 분리)
- 고객 신청 페이지(`/long-term/apply`)와 어드민 페이지(`/admin/long-term-bookings`)를 **반드시 분리**
- 고객 API는 어드민 인증 없이 동작하되, **`status='REQUESTED'` 로만 저장** (가격 필드 0)
- 신청 접수 시 관리자에게 **SMS + 이메일 동시 알림** (절대지침 1번 로깅 적용)
- 필요 env: `ADMIN_NOTIFY_PHONE`, `ADMIN_NOTIFY_EMAIL` (없으면 `STUDIO_PHONE`/`CONTACT_TO_EMAIL` fallback)

### 9-4. 통합 예약현황 캘린더 (선택 — 의뢰별)
- 의뢰처가 복수 예약 유형(룸/파티룸/장기대관)을 가질 경우, 단일 캘린더로 통합 노출 권장
- 읽기 전용 API 1개로 월 단위 조회: `GET /api/reservations/status?month=YYYY-MM`
- 카카오페이 심사 중 빌드라면 **SELECT만** 수행, 결제/예약 컬럼·라우트 변경 금지 (절대지침 10-10)

### 9-5. 클래스/레슨 공고 등록 권한 분리 원칙
- **공고 등록** = CM/ADMIN 전용, **수강 요청/신청** = MEMBER 전용 (CM/ADMIN의 신청 제출은 클라이언트+서버 모두에서 차단)
- 공고는 **실제 DB(`class_offerings`)에 저장** — localStorage 기반 모의 등록은 만들지 않는다 (다른 회원에게 노출되지 않아 운영 불가)
- 동일 폼 UI를 type prop으로 공유 (예: `AnnouncementsClient.tsx` 패턴) — 원데이클래스/레슨 코드 중복 방지
- 어드민 라우트와 공개 라우트는 **반드시 분리** (`/api/admin/class-offerings` ≠ `/api/class-offerings`)

### 9-6. CM 카드 분류별 공개 노출 원칙
- CM 신청 단계에서 **`can_oneday` / `can_lesson` 플래그**를 받아 어느 분류로 활동할지 결정 (둘 다 가능)
- CM 카드 노출은 두 위치: (1) 카테고리 메인 페이지 본문 (CmCardSection), (2) 전용 CM 목록 페이지 (`<category>/cm-list`)
- 두 위치 모두 `cm_applications.status='APPROVED'` + `can_<type>=true` 기준으로 자동 분류 — 관리자가 카테고리를 별도로 지정하지 않음
- 본문 임베드와 목록 페이지의 노출 토글은 **CM이 마이페이지에서 독립 제어** 가능하도록 컬럼 분리 (`show_in_section`, `show_in_list`) — A1STUDIO 검증 완료
- 레거시 `is_public`은 두 컬럼의 OR 로 자동 동기화. 신규 코드는 분리 컬럼만 참조
